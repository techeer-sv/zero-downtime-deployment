# 무중단 배포 전략의 3가지 유형

무중단 배포(Zero Downtime Deployment)란 서비스 중단 없이 새로운 버전의 애플리케이션을 배포하는 기법입니다.  
대표적인 전략은 세 가지입니다: **롤링 배포**, **블루-그린 배포**, **카나리 배포**.

---

## 1. 롤링 배포 (Rolling Deployment)

### 개념
기존 인스턴스를 한 번에 모두 교체하지 않고, **일부씩 순차적으로** 새 버전으로 교체하는 방식입니다.  
예를 들어 서버가 10대라면 2대씩 5번에 걸쳐 교체합니다.

### 동작 방식
```
[배포 전]  v1 v1 v1 v1 v1 v1 v1 v1 v1 v1
[1단계]    v2 v2 v1 v1 v1 v1 v1 v1 v1 v1
[2단계]    v2 v2 v2 v2 v1 v1 v1 v1 v1 v1
[3단계]    v2 v2 v2 v2 v2 v2 v1 v1 v1 v1
[완료]     v2 v2 v2 v2 v2 v2 v2 v2 v2 v2
```

1. 로드 밸런서에서 교체 대상 인스턴스를 트래픽에서 제외(drain)합니다.
2. 해당 인스턴스에 새 버전을 배포하고 헬스 체크를 통과하면 다시 트래픽에 추가합니다.
3. 위 과정을 모든 인스턴스가 교체될 때까지 반복합니다.

### 장점
- 추가 인프라 비용이 거의 없습니다 (기존 서버를 재사용).
- 설정이 단순하고 대부분의 플랫폼(Kubernetes, ECS 등)에서 기본 지원합니다.

### 단점
- **배포 중 v1과 v2가 동시에 서비스됩니다.** API 스펙이나 DB 스키마가 하위 호환되지 않으면 장애가 발생할 수 있습니다.
- 문제 발생 시 롤백 속도가 느릴 수 있습니다.
- 인스턴스 수가 적을수록 배포 중 처리 가능한 요청 수가 줄어듭니다.

### 적합한 상황
- 하위 호환성이 보장된 변경사항 (기능 추가, 버그 수정 등)
- 인프라 비용을 최소화해야 하는 경우
- Kubernetes Deployment의 기본 전략으로 널리 사용됩니다.

## 2. 블루-그린 배포 (Blue-Green Deployment)

### 개념
**완전히 동일한 두 개의 프로덕션 환경**을 구성합니다. 현재 서비스 중인 환경을 **Blue**, 새 버전을 배포할 환경을 **Green**이라 부릅니다.  
새 버전 준비가 완료되면 로드 밸런서의 트래픽을 Blue에서 Green으로 **한 번에** 전환합니다.

### 동작 방식
```
[배포 전]
  사용자 트래픽 → [Blue: v1] (활성)
                  [Green: v1] (유휴)

[Green에 v2 배포 후]
  사용자 트래픽 → [Blue: v1] (활성)
                  [Green: v2] (대기, 테스트 중)

[트래픽 전환]
  사용자 트래픽 → [Blue: v1] (유휴, 롤백 대비 유지)
                  [Green: v2] (활성)
```

1. Green 환경에 새 버전을 배포합니다.
2. Green 환경에서 통합 테스트, 스모크 테스트 등을 충분히 수행합니다.
3. 로드 밸런서 또는 DNS 레벨에서 트래픽을 Green으로 전환합니다.
4. 문제 발생 시 Blue로 즉시 롤백합니다. 이상 없으면 Blue 환경을 정리합니다.

### 장점
- 트래픽 전환이 순간적으로 이루어져 **v1과 v2가 동시에 서비스되는 시간이 없습니다.**
- 문제 발생 시 트래픽을 Blue로 되돌리기만 하면 되므로 **롤백이 매우 빠릅니다.**
- 배포 전 Green 환경에서 프로덕션과 동일한 환경으로 사전 검증이 가능합니다.

### 단점
- **인프라 비용이 2배**가 됩니다. 두 환경을 동시에 운영해야 하기 때문입니다.
- DB 스키마 변경이 있는 경우 Blue/Green 양쪽이 공유하는 DB를 어떻게 처리할지 별도로 설계해야 합니다.
- 전환 순간 진행 중이던 세션(stateful connection)이 끊길 수 있습니다.

### 적합한 상황
- 롤백 속도가 가장 중요한 미션 크리티컬 서비스
- 배포 전 프로덕션과 동일한 환경에서 철저한 검증이 필요한 경우
- 인프라 비용을 감당할 수 있는 경우

## 3. 카나리 배포 (Canary Deployment)

### 개념
새 버전을 전체 사용자에게 한 번에 노출하지 않고, **소수의 사용자에게 먼저** 트래픽을 흘려보내는 방식입니다.  
탄광에서 유독가스를 감지하기 위해 카나리아 새를 먼저 들여보냈던 것에서 유래했습니다.

### 동작 방식
```
[초기 단계: 5% 트래픽을 v2로]
  95% 사용자 → [v1 서버]
   5% 사용자 → [v2 서버] ← 모니터링 집중

[안정 확인 후: 50% 트래픽으로 확대]
  50% 사용자 → [v1 서버]
  50% 사용자 → [v2 서버]

[최종: 100% 전환]
  100% 사용자 → [v2 서버]
```

1. 전체 인스턴스의 일부(예: 5~10%)에만 새 버전을 배포합니다.
2. 에러율, 레이턴시, 비즈니스 지표 등을 집중적으로 모니터링합니다.
3. 지표가 정상이면 점진적으로 비율을 높여갑니다 (5% → 25% → 50% → 100%).
4. 문제가 발생하면 카나리 인스턴스만 롤백하여 전체 영향을 최소화합니다.

### 트래픽 분배 방법
- **가중치 기반(Weight-based):** 로드 밸런서에서 비율로 분배 (예: Nginx, AWS ALB, Istio)
- **사용자 기반(User-based):** 특정 사용자 그룹(내부 직원, 베타 유저)에게 먼저 노출
- **헤더/쿠키 기반:** 특정 헤더 값이 있는 요청만 카나리 버전으로 라우팅

### 장점
- **실제 프로덕션 트래픽으로 검증**하므로 스테이징 환경에서 잡지 못한 문제를 발견할 수 있습니다.
- 문제 발생 시 전체 사용자가 아닌 **소수의 사용자만 영향**을 받습니다.
- A/B 테스트와 결합하여 새 기능의 비즈니스 효과도 동시에 측정할 수 있습니다.

### 단점
- 배포 과정이 길고 복잡합니다. 지표 모니터링과 단계적 트래픽 전환 자동화 시스템이 필요합니다.
- 롤링과 마찬가지로 배포 중 **v1과 v2가 동시에 서비스**됩니다.
- 카나리 단계에서 발생한 데이터(DB 변경, 이벤트 등)를 어떻게 처리할지 고려해야 합니다.

### 적합한 상황
- 대규모 트래픽을 처리하는 서비스에서 위험을 최소화하며 배포할 때
- 새 기능의 성능 및 비즈니스 영향을 실측 데이터로 검증하고 싶을 때
- Argo Rollouts, Flagger, Spinnaker 등 점진적 배포 도구를 활용할 수 있을 때

## 전략 비교 요약

| 항목 | 롤링 배포 | 블루-그린 배포 | 카나리 배포 |
|------|-----------|---------------|-------------|
| 인프라 비용 | 낮음 | 높음 (2배) | 중간 |
| 롤백 속도 | 느림 | 매우 빠름 | 빠름 (카나리만 롤백) |
| 배포 중 두 버전 공존 | 있음 | 없음 | 있음 |
| 프로덕션 검증 | 불가 | 가능 (Green에서 사전 검증) | 가능 (실 트래픽) |
| 구현 복잡도 | 낮음 | 중간 | 높음 |
| 영향 범위 (문제 시) | 전체 | 전체 (전환 후) | 일부 사용자 |

---

> **선택 기준:** 비용이 제한적이라면 롤링, 빠른 롤백이 최우선이라면 블루-그린, 위험을 최소화하며 실 트래픽으로 검증하고 싶다면 카나리를 선택하세요.
