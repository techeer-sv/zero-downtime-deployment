# 카나리 배포 (Canary Deployment)

카나리 배포는 새 버전을 전체 사용자에게 한 번에 노출하지 않고, 트래픽을 **단계적으로 늘려가며** 안전하게 전환하는 방식이다.
각 단계마다 헬스체크와 안정화 대기(bake)를 거쳐 문제가 생기면 즉시 롤백한다.

## 업데이트 흐름

```
[구버전 100%] → [카나리 기동] → [5%] → [20%] → [50%] → [100%] → [구버전 종료]
                                  ↓       ↓        ↓        ↓
                               헬스체크  헬스체크  헬스체크  헬스체크
                              (실패 시) (실패 시) (실패 시) (실패 시)
                                  ↓       ↓        ↓        ↓
                               [자동 롤백 → 구버전 100% 복구]
```

## 사용법

```bash
bash scripts/deploy.sh <이미지_태그>

# 예시
bash scripts/deploy.sh v1.0.1
```

## 전체 흐름

```
[인자 확인] → [환경 감지] → [카나리 기동] → [5%→20%→50%→100% 단계 전환] → [구버전 종료]
```

---

## 환경 감지 → 배포 방향 결정

`docker ps` 로 실행 중인 `app-a` / `app-b` 컨테이너 수를 확인해서 배포 방향을 자동으로 결정한다.

| 현재 상태 | 안정 환경 | 카나리 환경 | 비고 |
|-----------|-----------|-------------|------|
| 둘 다 없음 | — | app-a | 초기 배포 (카나리 없이 바로 100%) |
| app-a만 실행 중 | app-a | app-b | 일반 카나리 배포 |
| app-b만 실행 중 | app-b | app-a | 일반 카나리 배포 |
| 둘 다 실행 중 | ❌ 오류 종료 | — | 이전 배포가 덜 끝난 상태 |

---

## 헬스체크

- `docker ps` 출력에서 `(healthy)` 상태 컨테이너 수를 **5초 간격**으로 폴링
- 목표 개수의 컨테이너가 모두 healthy 상태가 될 때까지 **최대 90초** 대기
- docker compose에 정의된 healthcheck (`GET /health`) 결과를 사용한다
- 타임아웃 시 자동 롤백 실행

---

## 카나리 단계별 전환 (app-b를 카나리로 배포하는 경우)

각 단계 사이에는 **2분(120초) bake** 대기 후 헬스체크를 수행한다.

| 단계 | 적용 템플릿 | app-b (카나리) | app-a (안정) | replica (b/a) | 설명 |
|------|------------|----------------|--------------|---------------|------|
| 1단계 | — | 기동 중 | 100% | 1 / 3 | 카나리 컨테이너 1개 기동 후 헬스체크 |
| 2단계 | `b-5.yaml` | 5% | 95% | 1 / 3 | 소량 트래픽으로 카나리 검증 |
| 3단계 | `b-20.yaml` | 20% | 80% | 2 / 3 | replica 증설 후 트래픽 확대 |
| 4단계 | `b-50.yaml` | 50% | 50% | 3 / 3 | 절반 트래픽, 완전한 검증 |
| 5단계 | `b-100.yaml` | 100% | 0% | 3 / 3 | 전체 전환, 구버전 트래픽 차단 |
| 정리 | `b-only.yaml` | 100% | — | 3 / 0 | 구버전(app-a) 라우터 제거 후 종료 |

### app-a를 카나리로 배포하는 경우 (app-b가 안정 환경일 때)

| 단계 | 적용 템플릿 | app-a (카나리) | app-b (안정) | replica (a/b) | 설명 |
|------|------------|----------------|--------------|---------------|------|
| 1단계 | — | 기동 중 | 100% | 1 / 3 | 카나리 컨테이너 1개 기동 후 헬스체크 |
| 2단계 | `a-5.yaml` | 5% | 95% | 1 / 3 | 소량 트래픽으로 카나리 검증 |
| 3단계 | `a-20.yaml` | 20% | 80% | 2 / 3 | replica 증설 후 트래픽 확대 |
| 4단계 | `a-50.yaml` | 50% | 50% | 3 / 3 | 절반 트래픽, 완전한 검증 |
| 5단계 | `a-100.yaml` | 100% | 0% | 3 / 3 | 전체 전환, 구버전 트래픽 차단 |
| 정리 | `a-only.yaml` | 100% | — | 3 / 0 | 구버전(app-b) 라우터 제거 후 종료 |

## 롤백

헬스체크 실패 시 자동으로 롤백한다.

```
[헬스체크 실패]
      ↓
{stable}-100.yaml 적용  →  안정 환경 100% 트래픽 복구
      ↓  (10초 대기)
{stable}-only.yaml 적용  →  카나리 환경 라우터에서 제거
      ↓
카나리 컨테이너 종료  →  롤백 완료
```

## 템플릿 파일

`templates/` 폴더의 yaml 파일을 `dynamic/canary-routers.yaml`에 덮어쓰는 방식으로 Traefik 라우터를 전환한다.

### app-b 카나리용 템플릿

| 파일 | app-b 가중치 | app-a 가중치 | 의미 |
|------|-------------|-------------|------|
| `b-5.yaml` | 5% | 95% | 카나리 5% 전환 |
| `b-20.yaml` | 20% | 80% | 카나리 20% 전환 |
| `b-50.yaml` | 50% | 50% | 카나리 50% 전환 |
| `b-100.yaml` | 100% | 0% | 카나리 100% 전환 (구버전 트래픽 차단) |
| `b-only.yaml` | 100% | — | 구버전 라우터에서 완전 제거 |

### app-a 카나리용 템플릿

| 파일 | app-a 가중치 | app-b 가중치 | 의미 |
|------|-------------|-------------|------|
| `a-5.yaml` | 5% | 95% | 카나리 5% 전환 |
| `a-20.yaml` | 20% | 80% | 카나리 20% 전환 |
| `a-50.yaml` | 50% | 50% | 카나리 50% 전환 |
| `a-100.yaml` | 100% | 0% | 카나리 100% 전환 (구버전 트래픽 차단) |
| `a-only.yaml` | 100% | — | 구버전 라우터에서 완전 제거 |

## 파일 구조

```
plans/canary/
├── a.yaml                     # app-a docker compose 파일 (port 8080-8082)
├── b.yaml                     # app-b docker compose 파일 (port 8083-8085)
├── docker-compose.yaml        # Traefik + 공통 네트워크 설정
├── dynamic/
│   └── canary-routers.yaml    # Traefik이 감시하는 동적 라우터 설정 (배포 중 교체됨)
├── templates/
│   ├── a-5.yaml               # app-a 5% 카나리
│   ├── a-20.yaml              # app-a 20% 카나리
│   ├── a-50.yaml              # app-a 50% 카나리
│   ├── a-100.yaml             # app-a 100% 전환
│   ├── a-only.yaml            # app-a만 라우터에 등록
│   ├── b-5.yaml               # app-b 5% 카나리
│   ├── b-20.yaml              # app-b 20% 카나리
│   ├── b-50.yaml              # app-b 50% 카나리
│   ├── b-100.yaml             # app-b 100% 전환
│   └── b-only.yaml            # app-b만 라우터에 등록
└── scripts/
    ├── deploy.sh              # 메인 배포 스크립트
    ├── deploy-a.sh            # app-a 컨테이너 기동
    ├── deploy-b.sh            # app-b 컨테이너 기동
    ├── scale-a.sh             # app-a replica 수 조정
    ├── scale-b.sh             # app-b replica 수 조정
    ├── down-a.sh              # app-a 컨테이너 종료
    └── down-b.sh              # app-b 컨테이너 종료
```
